# 再塑通 RePlasMatch — 微信云托管迁移指南

---

## 概述

将再塑通后端从 ECS (Alibaba Cloud Linux) 迁移到**微信云托管（CloudBase Run）**。

### 迁移收益

| 对比项 | ECS 自建 | 微信云托管 |
|--------|----------|-----------|
| HTTPS/SSL | 需自配 Nginx + Let's Encrypt | **内置免费 SSL** |
| 域名 | 需自购 + ICP 备案 | **自动分配 `*.run.tcloudbase.com`** |
| 数据库 | SQLite 单文件（无备份） | **TDSQL-C MySQL（自动备份）** |
| 文件存储 | 本地磁盘（无容灾） | **CloudBase COS（99.9% 持久性）** |
| 扩缩容 | 手动（PM2 单进程） | **自动弹性伸缩（1-3 实例）** |
| 日志/监控 | 手动查看 logs/ | **控制台可视化** |
| 微信集成 | 需手动配白名单 | **天然内网互通** |
| 运维 | 需维护 Nginx/PM2/OS | **零运维** |

router.get('/list', async (req, res) => {

### 3.3 快速异步化方案

提供一个批量改造脚本：

```bash
# 创建 bash 脚本批量替换（示例 — 需要验证）
cd 撮合小程序/backend/routes

# 1. 路由处理函数加上 async
for f in *.js; do
  # 替换 router.get/post/patch/delete 的回调为 async
  sed -i 's/router\.\(get\|post\|patch\|delete\|put\)((req, res)/router.\1(async (req, res)/g' "$f"
done

# 2. db 调用前加 await（注意：只对 .get/.all/.run/.exec 加）
# 此步骤需要人工审核，因为不是所有 db 调用都需要 await
# 例如 middleware 中的 db 调用如果在同步上下文则不能加
```

**⚠️ 不建议用 sed 全自动处理**，建议逐个文件手工改造，确保：
1. 所有包含 DB 调用的 Express handler 都是 `async function`
2. 所有 `db.prepare().get/run/all` 都加了 `await`
3. `db.exec()` 加了 `await`
4. `db.transaction()` 加了 `await`
5. 错误处理（try-catch）正常工作

## 四、文件存储迁移

### 4.1 改造 Vision 路由（示例）

COS 模式下，multer 使用 `memoryStorage`，上传后需手动调用 `storage.upload()`：

```javascript
// vision.js 中的改动
const storage = require('../storage/index');

// 使用存储适配器的 multer 配置
const upload = multer({
  storage: storage.getMulterStorage(),  // 本地: diskStorage, COS: memoryStorage
  limits: { fileSize: 10 * 1024 * 1024 },
  fileFilter: ...,
});

// 上传完成后保存
router.post('/analyze', upload.single('image'), async (req, res) => {
  try {
    const file = req.file;
    let imageUrl;

    if (storage.STORAGE_TYPE === 'cos') {
      // COS 模式：从内存 buffer 上传
      imageUrl = await storage.upload(
        file.buffer,
        file.originalname,
        file.mimetype
      );
    } else {
      // 本地模式：文件已在磁盘
      imageUrl = `/uploads/${file.filename}`;
    }

    // ... 后续 AI 识别逻辑
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});
```

## 十、首次部署排障

### 常见问题 1: JWT_SECRET 未设置导致容器崩溃

**现象**:
```
❌ FATAL: JWT_SECRET 环境变量未设置！
```

**原因**: `.dockerignore` 排除了 `.env` 文件，且 `container.config.json` 中未配置 `JWT_SECRET`。

**已修复**: 
- `middleware/auth.js` 不再 `process.exit(1)`，而是 `crypto.randomBytes(64)` 自动生成密钥（带警告）
- 生产环境建议在云托管控制台手动设置 `JWT_SECRET` 和 `REFRESH_SECRET`

### 常见问题 2: DB_TYPE=mysql 但未配置 MySQL

**现象**: 容器无法连接到 MySQL，不断重启。

**修复**: 首次部署时 `container.config.json` 中 `DB_TYPE=sqlite`，配合云硬盘挂载持久化数据。
切换到 MySQL 后再改环境变量。

### 常见问题 3: docker build 缓存导致旧代码未更新

**解决**: 在云托管控制台 → 版本管理 → 重新构建时勾选「不使用缓存」。

### 常见问题 4: 健康检查失败（端口不对）

云托管网关默认探活 `/` 而非 `/api/health`。可通过 `container.config.json` 配置：
```json
{
  "livenessProbe": {
    "httpGet": { "path": "/api/health", "port": 3456 }
  },
  "readinessProbe": {
    "httpGet": { "path": "/api/health", "port": 3456 }
  }
}
```

## 十一、FAQ

**Q: SQLite 和 MySQL 能共存吗？**
A: 可以。`DB_TYPE=sqlite` 时用原 SQLite，`DB_TYPE=mysql` 时用 MySQL。DB factory 自动选择。

**Q: WebSocket 在云托管支持吗？**
A: 完全支持。云托管网关自动处理 WebSocket 升级。单实例无需 Redis。

**Q: 迁移需要停机吗？**
A: 建议在低峰期进行。DNS 切换后有几分钟生效延迟。可先并行运行，验证无误后再切流量。

**Q: admin.html 中的 API 地址怎么办？**
A: 需要改为云托管域名。建议改为相对路径：`const API = '/api'`（同域部署时自动生效）。

**Q: 容器重启后 JWT token 会失效吗？**
A: 如果使用自动生成的密钥，是的，所有用户需重新登录。在云托管控制台设置 `JWT_SECRET` 环境变量即可持久化。
